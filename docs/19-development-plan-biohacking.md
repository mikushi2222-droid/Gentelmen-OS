## 0. Где мы сейчас (факт, не пожелания)

Что **уже есть** в коде (проверено по `lib/`):

- **Биохакинг (Уровень 3), domain:** `optimization.dart` (`optimizationScore`,
  `bottlenecks`, `maxImpactActions` с `explanation`), `evidence.dart`
  (`EvidenceRating` A/B/C), `knowledge_base.dart` (сид-протоколы `kProtocols` и
  добавки `kSupplements`). Экран `biohacking_screen.dart`.
- **Здоровье (Уровень 2):** `health_marker.dart` (17 типов маркеров с
  референсами/единицами/статусом), `biological_age.dart`, экраны обзора и детали,
  `health_ai_analyzer.dart` — ИИ-разбор маркеров через RouterAI **с веб-поиском**.
- **Router AI:** `router_ai_client.dart` — клиент chat/vision, **подробное
  логирование с маскировкой ключа**, веб-поиск (`plugins: web`), JSON-режим,
  `analyzeImage` (мультимодаль). `router_ai_advisor.dart` с **оффлайн-фолбэком**.
- **Логирование:** `app_logger.dart` (кольцевой буфер 500 записей, уровни,
  экспорт в `.md`), экран `log_screen.dart` (`/settings/logs`), глобальный
  перехват ошибок и зоны в `main.dart`.

Чего **нет** (и это и есть scope этого плана):

- Нет персистентности биохакинга: таблиц `LabResults`, `Supplements`,
  `SupplementIntakes`, `Protocols`/`ProtocolSteps`, `SleepLogs` — добавки и
  протоколы пока только константы в коде.
- Нет раздела **лёгкого спорта** (йога, растяжка, мобильность) вообще.
- Нет **загрузки/расшифровки анализов** через Router AI (фото бланка → маркеры).
- Нет **исследований доказательности** через веб-поиск как отдельного,
  кэшируемого и объяснимого потока (есть только разовый текстовый разбор).
- **Логирование действий неполное:** залогированы ошибки, навигация и ИИ, но
  большинство пользовательских мутаций (привычки, маркеры, XP, миссии, образы,
  покупки, профиль) на экран отладки **не пишут** — см. §6.

## 1. Приоритеты направления (по CLAUDE.md: приватность → скорость → объяснимость → простота)

Биохакинг — **система принятия решений**, а не склад БАДов. Каждый новый экран
отвечает на вопрос «что ограничивает прогресс и что даст максимум эффекта», даёт
`explanation`, работает офлайн, ИИ — только поверх с фолбэком, дисклеймер «не
медицина» — на каждом экране здоровья/биохакинга.

Пять рабочих потоков (workstreams), реализуются по фазам:

| WS | Тема | Суть |
|----|------|------|
| A | **Анализы + расшифровка** | таблица `LabResults`, ввод вручную, **фото бланка → маркеры через Router AI**, графики/зоны, история по годам |
| B | **Добавки + приём** | `Supplements`/`SupplementIntakes`: справочник (сид) → чек-лист «Сегодня», связь добавка↔дефицит маркера |
| C | **Доказательность (research)** | объяснимый веб-поиск Router AI по добавке/протоколу, кэш `EvidenceNotes`, рейтинг A/B/C с источниками |
| D | **Лёгкий спорт** | `MobilityRoutines`/`MobilitySessions`: йога, растяжка, мобильность — сид-комплексы, трекинг, влияние на «Дисциплину»/«Биохакинг» |
| E | **Логирование действий** | каждое действие пользователя и мутация репозитория пишут запись на экран отладки (§6) |

## 3. Фаза V2.7 — Анализы и расшифровка через Router AI (WS A)

Самый ценный дифференциатор. Делится на под-инкременты по слоям.

### 3.1 drift: `LabResults` (schemaVersion 5 → 6)
- Таблица `LabResults` с `@DataClassName('LabResultsData')`: `id`, `markerType`
  (строка-код `HealthMarkerType`), `value` (double), `unit`, `takenAt` (дата
  забора), `source` (manual/ai), `note`. Индекс по `(markerType, takenAt)`.
- Поднять `schemaVersion` до 6, дописать шаг `onUpgrade` (создание таблицы),
  добавить **миграционный тест** (путь v1→v6 без дыр).
- DAO: `insert`, `latestByType`, `historyByType` (одним запросом, без N+1),
  `byYear`. Маппер строк ↔ доменная модель.

### 3.2 domain (чистые функции, без Flutter/drift)
- Переиспользуем `markerStatus`/референсы из `health_marker.dart` для зон
  «Оптимально/Норма/Низко/Высоко».
- `labTrend(history)` — направление и дельта по годам (объяснимо), `byYearSeries`
  для графика. Unit-тесты в `test/unit/`.

### 3.3 Расшифровка фото бланка через Router AI (ключевая фича)
- Поток: `image_picker` → копия в documents (`persistWardrobeImage`-аналог для
  анализов) → base64 → `RouterAiClient.analyzeImage` с промптом, требующим
  **строгий JSON**: массив `{markerCode, value, unit, takenAt, confidence}`.
- Новый класс `lib/features/health/domain/lab_ocr_parser.dart` (без Flutter):
  валидирует/нормализует ответ ИИ в список доменных `LabResultDraft`
  (сопоставление синонимов названий → `HealthMarkerType`, проверка единиц).
- Оркестрация в `application/` через провайдер; **оффлайн-фолбэк** — ручной ввод
  (ИИ недоступен → открываем форму с пустыми полями, приложение не падает).
- UI: экран «Загрузить анализ» → превью распознанных строк → пользователь
  подтверждает/правит → сохранение в `LabResults`. Дисклеймер «распознавание
  черновое, проверьте значения; интерпретирует врач».
- **Логирование (тег `Labs`):** выбор фото, отправка в Router AI (размер b64),
  код/время ответа (уже логируется в клиенте), число распознанных маркеров,
  подтверждение/правка, ошибки и фолбэк.
- **Приватность:** фото анализа уходит в Router AI **только по явному действию**
  пользователя (кнопка «Распознать»), с предупреждением, что данные отправятся
  в облако; по умолчанию — ручной ввод, ничего в сеть.

### 3.4 Текстовый ИИ-разбор результатов
- Переиспользуем `HealthAiAnalyzer.analyze` (уже есть, web-search) поверх
  сохранённых `LabResults`: «что не в норме и что с этим делать» — образовательно,
  со ссылками на источники, с фолбэком.

**DoD:** можно внести анализ вручную и распознать с фото; история рисуется
графиком с зонами; разбор работает офлайн (фолбэк) и онлайн; все шаги в журнале.

## 4. Фаза V2.8 — Добавки и приём (WS B) + Исследования доказательности (WS C)

### 4.1 drift: `Supplements` + `SupplementIntakes` (schemaVersion 6 → 7)
- `Supplements` (`@DataClassName('SupplementsData')`): справочник — `name`,
  `evidence` (A/B/C), `dose`, `risks`, `interactions`, `targetMarkers` (какие
  дефициты адресует). **Сид** из текущего `kSupplements` в `onCreate` +
  идемпотентный `insertOnConflictUpdate` (не пользовательские данные).
- `SupplementIntakes` (`@DataClassName('SupplementIntakesData')`): факт приёма
  (`supplementId`, `takenAt`) — для чек-листа «Сегодня» и стриков.
- Поднять `schemaVersion` → 7 + `onUpgrade` + миграционный тест.

### 4.2 Связь «дефицит → добавка» (domain)
- Чистая функция `suggestSupplements(markers)`: по маркерам не в норме
  предлагает добавки из справочника с `explanation` («Витамин D 18 нг/мл < 30 →
  D3 2000–4000 МЕ»). Без магазина и партнёрских ссылок (анти-цель §10 видения).

### 4.3 Исследования доказательности (WS C, Router AI web-search)
- Класс `lib/features/biohacking/domain/evidence_research.dart` (оркестрация в
  `application/`): по добавке/протоколу делает Router AI chat **с `webSearch:
  true`**, просит структурированный JSON `{rating: A/B/C, summary, sources[]}`.
- Кэш в новой таблице `EvidenceNotes` (`@DataClassName('EvidenceNotesData')`):
  `subject`, `rating`, `summary`, `sourcesJson`, `fetchedAt` — чтобы не дёргать
  сеть повторно и работать офлайн (показываем закэшированное + дату).
- **Объяснимость:** рейтинг A/B/C всегда сопровождается источниками и кратким
  обоснованием; при отсутствии сети — показываем сид-рейтинг из `evidence.dart`.
- **Логирование (тег `Research`/`Biohacking`):** запрос, web-поиск вкл/выкл,
  код/время ответа, итоговый рейтинг, попадание в кэш, фолбэк.

**DoD:** карточка добавки показывает дозировку/риски/взаимодействия + рейтинг с
источниками (онлайн) или сид-рейтинг (офлайн); приём добавки попадает в чек-лист
«Сегодня» и в журнал.

## 5. Фаза V2.9 — Лёгкий спорт: йога, растяжка, мобильность (WS D)

Не фитнес-зал и не силовые (они в `fitness/`), а **мягкая мобильность** как часть
биохакинга (восстановление, сон, стресс, осанка).

### 5.1 drift: `MobilityRoutines` + `MobilitySessions` (schemaVersion 7 → 8)
- `MobilityRoutines` (`@DataClassName('MobilityRoutinesData')`): сид-комплексы —
  `name`, `kind` (yoga/stretch/mobility), `durationMin`, `goal` (сон/осанка/
  восстановление), `stepsJson` (последовательность с длительностью). Сид в
  `onCreate`.
- `MobilitySessions` (`@DataClassName('MobilitySessionsData')`): факт выполнения
  (`routineId`, `completedAt`, `durationMin`) — для стриков и под-оценок.
- Поднять `schemaVersion` → 8 + `onUpgrade` + миграционный тест.

### 5.2 Сид-контент (на русском, спокойный тон)
- 4–6 коротких комплексов: «Утренняя мобильность 7 мин», «Растяжка спины после
  сидячего дня», «Йога для сна 10 мин», «Раскрытие тазобедренных», «Шея/плечи за
  компьютером». Каждый — последовательность с подсказками, без «магии» и без
  медицинских обещаний (дисклеймер про противопоказания).

### 5.3 domain + UI
- Чистая функция вклада мобильности в под-оценки «Дисциплина»/«Биохакинг»
  (объяснимо). Экран списка комплексов → запуск с таймером шага → отметка
  выполнения. `TextEditingController`/таймеры — диспозить.
- **Логирование (тег `Mobility`):** старт комплекса, завершение, длительность.

**DoD:** можно открыть комплекс, пройти по шагам, отметить выполнение; сессия в
истории и в журнале; вклад в под-оценки виден на Главной.

## 6. Сквозной поток E — Логирование всех действий (Definition: «видно на экране отладки»)

Цель: **любое значимое действие пользователя и любая мутация данных** оставляет
запись в `AppLogger`, видимую на `/settings/logs` (и в экспортном `.md`).

Текущее покрытие (факт): логируются ошибки/зоны (`main.dart`), навигация
(`app_router.dart`), весь Router AI (`router_ai_client`/`router_ai_advisor`),
ИИ-анализ здоровья, ИИ-фичи гардероба. **Не логируются** мутации в:

| Файл | Что добавить (тег) |
|------|--------------------|
| `habits/application/habits_providers.dart` | отметка/снятие привычки, стрик (`Habits`) |
| `health/application/health_providers.dart` | добавление/правка маркера (`Health`) |
| `fitness/application/fitness_providers.dart` | новый замер (`Fitness`) |
| `rpg/application/rpg_providers.dart` | начисление XP, уровень, разблокировка ачивки (`Rpg`) |
| `dashboard/application/dashboard_providers.dart` | генерация/выполнение миссии дня (`Missions`) |
| `outfit_builder/application/outfit_providers.dart` | генерация/сохранение/оценка образа (`Outfit`) |
| `purchases/application/purchases_providers.dart` | добавление/закрытие желания покупки (`Purchases`) |
| `profile/data/profile_repository_impl.dart` | сохранение профиля (`Profile`) |
| `wardrobe/.../wardrobe_repository_impl.dart` | CRUD вещи, отметка носки (`Wardrobe`) |
| `settings/application/settings_providers.dart` | смена темы, ключа Router AI (**маскировать!**), модели (`Settings`) |

Правила:
- `log.i(tag, 'человекочитаемое действие + ключевой параметр')` на успех,
  `log.w/e` на ошибку с `error:`/`stackTrace:`.
- **Секреты не логируем**: ключ Router AI — только маскированно (как в
  `_maskKey`); тело здоровья/анализов — без полных значений в проде, только факт
  действия и количество.
- Без синхронного I/O в `build()`; логирование — в слое
  application/data, не в виджетах.

**DoD:** ручной прогон ключевых сценариев → на экране отладки видно по записи на
каждое действие; релизный уровень `info` сохраняет важное; ничего секретного.

## 8. Инварианты (нарушение = переделка), напоминание из CLAUDE.md

- drift: каждой таблице `@DataClassName('<Таблица>Data')`; при изменении схемы —
  поднять `schemaVersion` (от 5) и дописать `onUpgrade` + миграционный тест.
- Бизнес-логика — чистыми функциями в `domain/` (без Flutter/drift), с
  `explanation`; оркестрация — в `application/` (Riverpod); UI — в `presentation/`.
- Router AI: **всегда оффлайн-фолбэк**; ключ — только secure storage,
  маскировать в логах; данные в сеть — только по явному действию пользователя.
- `Color.withValues(alpha:)`, не `withOpacity`; `Image.file(errorBuilder:)`, не
  `existsSync()` в `build`; контроллеры/таймеры — диспозить; календарные даты —
  `DateTime(y,m,d-1)`.
- Коды достижений — только из `Achv` (`achievement_catalog.dart`); каждая
  засеянная ачивка имеет реальный триггер.
- Перед коммитом: `flutter pub get` → `build_runner build` → `flutter analyze`
  (0/0) → `flutter test` (зелёные).
