# 05. Движок рекомендаций

Это одна из ключевых частей. Строим её **без «магии» и без чёрного ящика** —
чистыми функциями на основе понятных правил. Любая рекомендация **объяснима**.

## 1. Где живёт логика

Движок — это чистый Dart в `features/outfit_builder/domain/`. Никаких
зависимостей от Flutter/Drift — тривиально тестируется (`test/unit/outfit/`,
`test/unit/recommendation/`).

```
features/outfit_builder/domain/
├── outfit_scorer.dart        // итоговый scoring образа (включая комфорт)
├── fit_rules.dart            // правила посадки под фигуру
├── color_harmony.dart        // цветовая гармония
├── occasion_rules.dart       // соответствие поводу/дресс-коду
├── weather_rules.dart        // соответствие погоде/сезону
└── outfit_generator.dart     // генерация кандидатов из гардероба
```

> Отдельного `comfort_rules.dart` нет: комфортная ось считается внутри
> `outfit_scorer.dart` (`comfortScore`: база 0.6 + бонус `(rating-3)*0.1` за вещь).

## 2. Четыре (пять) осей оценки

Каждый комплект получает score по осям, итог — взвешенная сумма:

| Ось | Поле | Что оценивает |
|-----|------|---------------|
| Посадка | `fitScore` | подходит ли крой фигуре пользователя |
| Цвет | `colorScore` | гармония цветов вещей между собой и с предпочтениями |
| Повод | `occasionScore` | соответствие поводу и дресс-коду |
| Погода | `weatherScore` | соответствие погоде/температуре/сезону |
| Комфорт | `comfortScore` | материалы и личные оценки удобства |

```
total = w_fit*fitScore + w_color*colorScore + w_occasion*occasionScore
      + w_weather*weatherScore + w_comfort*comfortScore
```

Веса по умолчанию (сумма = 1.0), настраиваемые:
```
w_fit = 0.30
w_occasion = 0.25
w_weather = 0.20
w_color = 0.15
w_comfort = 0.10
```

Каждая ось нормирована в [0, 1]; total масштабируется в 0..100 для UI.

## 3. Правила посадки (fit) под фигуру

Для мужчины с крупной фигурой базовые эвристики (рабочие, не «абсолютная истина»):

- лучше работают **прямые и структурированные** формы;
- лучше работает **средняя/высокая посадка** брюк;
- лучше работают **спокойные цвета**;
- лучше работают **плотные ткани**;
- избегать слишком **тонких, обтягивающих и блестящих** материалов.

Реализация (псевдокод):
```dart
double fitScore(ClothingItem item, UserProfile p) {
  var s = 0.6; // базовая
  final large = p.waist >= 110 || p.weight >= 100; // порог из профиля
  if (large) {
    if (item.fit == Fit.slim) { s -= 0.4; explain('skinny fit не подходит крупной фигуре'); }
    if (item.fit == Fit.regular || item.fit == Fit.straight || item.fit == Fit.comfort) {
      s += 0.3; explain('структурированный крой работает на крупной фигуре');
    }
    if (isTrousers(item) && item.rise == Rise.midOrHigh) {
      s += 0.2; explain('средняя/высокая посадка под большую талию');
    }
    if (isShinyOrThin(item.material)) { s -= 0.2; explain('тонкая/блестящая ткань нежелательна'); }
  }
  return s.clamp(0, 1);
}
```

> Это **эвристика как базовое правило, а не жёсткий закон.** Пороги берутся из
> профиля, веса настраиваемы, объяснения всегда прикладываются.

## 4. Цветовая гармония (colorScore)

- базовая палитра пользователя (`colorPreference`) повышает score;
- нейтральные базы (navy, серый, чёрный, белый, бежевый) — высокий бонус
  совместимости;
- проверка контраст/тон между верхом и низом;
- штраф за конфликтные сочетания (например, два ярких насыщенных принта).

Подход: цвета конвертируются в HSL; оцениваются по правилам
(нейтраль+нейтраль, нейтраль+акцент, аналоговые, комплементарные с осторожностью).

## 5. Соответствие поводу (occasionScore)

- сопоставление `DressCode` образа с заявленным поводом (матрица соответствия);
- каждая вещь имеет «формальность» (derived из категории/материала);
- среднее формальности образа сравнивается с целевой формальностью повода;
- штраф за рассогласование (например, кроссовки на business).

Матрица (упрощённо):
```
Occasion.formal      → требуется DressCode.business / blackTie
Occasion.work        → businessCasual / business
Occasion.smartCasual → smartCasual
Occasion.everyday    → casual / smartCasual
Occasion.sport       → casual (спорт-категории)
```

## 6. Соответствие погоде/сезону (weatherScore)

- `Season` вещи против текущего сезона/температуры;
- температурные пороги (например, пальто при < 8 °C, лёгкие ткани при > 24 °C);
- штраф за несезонность; бонус за «all-season».

## 7. Комфорт (comfortScore)

- личные оценки удобства вещей (`rating` 1..5) → нормируются;
- материалы: плотные/натуральные — бонус, синтетика/блестящее — штраф для крупной фигуры;
- учёт `restrictions` из профиля (запреты/неудобства) — жёсткий штраф или
  исключение вещи.

## 8. Генерация кандидатов (outfit_generator)

Алгоритм (для повода/погоды):
1. Отфильтровать доступные вещи (`isAvailable == true`) по сезону/категории.
2. Сгенерировать валидные комбинации по «слотам» образа:
   - верх (shirt/polo/tShirt) + низ (trousers/jeans) + обувь (+ опционально
     верхний слой blazer/jacket/coat + аксессуар).
3. Для каждой комбинации посчитать `OutfitScore` всеми осями.
4. Отсортировать по `total`, применить разнообразие (не возвращать 3 почти
   одинаковых), вернуть топ-1..3.
5. Приложить `explanation` к каждому.

Оптимизация: ограничить перебор (greedy/beam search), если гардероб большой —
сначала выбрать лучший низ/верх по отдельным осям, затем комбинировать.

## 9. Объяснимость (обязательное требование)

Каждый score сопровождается списком строк-объяснений:
```
explanation: [
  "Regular fit подходит вашей фигуре (+)",
  "Navy + серый — нейтральная гармония (+)",
  "Подходит для повода 'работа' (+)",
  "Сезон: осень — соответствует (+)",
  "Высокий рейтинг удобства 5/5 (+)"
]
```

Это строится прямо в правилах (каждое правило, меняющее score, добавляет строку).

## 10. Производные рекомендации профиля

Отдельно от образов: блок «что подходит вашей фигуре» на экране профиля.
Генерируется теми же правилами, но в виде советов:
- предпочтительные кройки;
- рекомендованная посадка брюк;
- ширина лацканов;
- предпочтительные ткани;
- чего избегать.

## 11. Настраиваемость

- Веса осей сейчас заданы константами в `outfit_scorer.dart` (вынос в Settings —
  возможное будущее улучшение).
- Пороги (что считать «крупной фигурой», температурные границы) — константы рядом
  с соответствующим правилом (`fit_rules.dart`, `weather_rules.dart`), а не в
  выделенном `core/constants/`.

## 12. Тестирование движка

- Юнит-тесты на каждое правило (fit/color/occasion/weather/comfort).
- Тесты на генератор: корректные слоты, отсутствие недоступных вещей.
- Property-тесты: total всегда в [0, 100], объяснение непустое если score изменён.
- Golden-кейсы: фиксированный гардероб + повод → ожидаемый топ-образ.
