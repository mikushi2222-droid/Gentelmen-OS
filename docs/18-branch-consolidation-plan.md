# 18. План консолидации веток в одну релизную линию

> Цель: оставить **одну актуальную релизную ветку — `main`** — со всеми нужными
> фичами, а все остальные ветки закрыть. Снимок и анализ — **2026-06-23**.
> Контекст по веткам — см. [17-branches-status.md](17-branches-status.md).

---

## 1. Конечное состояние (Definition of Done)

- `main` — единственная долгоживущая ветка, релизная линия, CI зелёный
  (analyze 0/0, тесты, сборка APK debug/release).
- Все ценные фичи из расходящихся веток **перенесены** в `main`.
- Все ветки `claude/*` удалены (локально и на origin).
- Дальше работаем по схеме «короткая feature-ветка → PR в `main` → удалить».

## 2. Почему нельзя просто слить (merge/rebase)

Анализ показал **две независимые истории**, реализующие пересекающиеся фичи:

| Фактор | `main` | `gentleman-os-plan-cgm6hh` (суперсет) |
|--------|--------|----------------------------------------|
| Общий предок | `fb53492` | `fb53492` |
| schemaVersion БД | **5** (миграции v1→v5) | **7** (другая линия v6/v7) |
| База | чистый пересбор: analyze 0/0/0, CLAUDE.md, drift-naming, CI закалён | старая база до этой чистки |
| Wear forecast (V2.4) | реализован заново (ткань+статус) | реализован иначе (`74946ff`) |
| Health/Биохакинг домен | `090e0bf` (палитра + ядро) | свой вариант |
| Dashboard под-оценки/«Совет дня» | V2.1 (`ea7a66e`) | свой вариант |
| custom_lint/riverpod_lint | удалены (`01e7a74`) | удалены (`ffa74c2`) |

**Вывод:** ветки не находятся в отношении «предок-потомок» по фичам — это
**параллельные реализации**. `git merge` дал бы конфликты в схеме БД (v5 ↔ v7),
в дублирующихся доменных функциях, в pubspec и доках практически в каждом файле.
Линии миграций несовместимы: импортировать чужие v6/v7 поверх нашей v5 нельзя —
получится разъехавшаяся схема у существующих пользователей.

> **Решение: не мёржим историю. Переносим недостающие фичи как свежие
> инкременты поверх чистого `main`** — по одной фиче на коммит/ветку, с
> продолжением нашей линии миграций (v5 → v6 → …), тестами и analyze 0/0.
> Источник для переноса — единственный суперсет `gentleman-os-plan-cgm6hh`
> (две другие расходящиеся ветки в него входят, разбирать их отдельно не нужно).

## 4. Пошаговый план

### Фаза 0. Зафиксировать текущий инкремент
- Довести **PR #4** (`review-and-fix-action-thycg1`) до зелёного CI и влить в
  `main` (журнал отладки, экспорт `.md`, износ на карточке, доки).
- После мёржа `main` — единственная актуальная точка отсчёта.

### Фаза 1. Подготовка
- `git fetch --all`, тег-снимок старых веток на случай отката:
  `git tag archive/plan-cgm6hh origin/claude/gentleman-os-plan-cgm6hh` (тег
  сохраняет коммиты даже после удаления веток).
- Сверить инвентаризацию (§3) с реальным `main`: для каждой 🔁 решить
  «переносим / уже есть / устарело».

### Фаза 2. Перенос фич (итеративно, по одной)
Для каждой выбранной фичи:
1. Ветка от свежего `main`: `git switch -c feat/<slug>`.
2. Изучить исходную реализацию в суперсете
   (`git show <commit> -- <files>`), **переписать под текущую архитектуру и
   CLAUDE.md** (drift `@DataClassName`, миграция от текущей версии `main`,
   `withValues`, чистые domain-функции, тесты). Не cherry-pick «как есть» —
   база разная.
3. Если фича требует таблиц/полей — **поднять schemaVersion от значения в
   `main` (5 → 6 → …)** и дописать `onUpgrade`; не переиспользовать чужие
   номера v6/v7.
4. `flutter analyze` 0/0 + `flutter test` зелёные.
5. PR в `main`, мёрж, удалить feature-ветку.

Рекомендуемый порядок (от низкого риска к высокому): 1 → 2 → 4 → 6 → 3 → 5 →
7 → 8 → 9 → 10 → 11. Сначала бесконфликтный UI/domain, БД-затрагивающие — позже,
по одной миграции за PR.

### Фаза 3. Сверка документации и схемы
- Обновить `07-roadmap.md` (отметить перенесённые фазы), `03-data-model.md`
  (итоговая схема), `17-branches-status.md` (ветки удалены).
- Убедиться, что финальный `schemaVersion` и `onUpgrade` покрывают путь от v1
  до текущего без дыр; добавить миграционный тест.

### Фаза 4. Финальная проверка
- `flutter pub get && dart run build_runner build --delete-conflicting-outputs`
- `flutter analyze` 0/0 · `flutter test` зелёные · сборка APK debug/release.

### Фаза 5. Уборка веток
После того как все нужные фичи в `main`:

```bash
# обсолетные (уже в main)
git push origin --delete claude/code-review-plk01k
git push origin --delete claude/project-code-review-c6vcdj
# расходящиеся (после переноса фич; коммиты сохранены в archive/* тегах)
git push origin --delete claude/gentleman-os-plan-cgm6hh
git push origin --delete claude/documentation-review-rdnwur
git push origin --delete claude/garment-wear-forecast-card-5o5oe8
# рабочая — после мёржа PR #4
git push origin --delete claude/review-and-fix-action-thycg1
# локальная чистка
git fetch --prune
```

## 5. Риски и митигации

| Риск | Митигация |
|------|-----------|
| Конфликт схемы БД (v5 ↔ v7) | Не импортировать чужие миграции; продолжать линию `main` заново, по одной миграции за PR + миграционный тест |
| Дубль уже сделанных в `main` фич | Шаг сверки 🔁 в §3 до переноса; не плодить второй wear-forecast/health-домен |
| Большой объём (V2.7–V2.11) | Дробить на под-инкременты по слоям, отдельный PR на фичу — каждый проходит CI |
| Потеря старого кода после удаления веток | `archive/*` теги фиксируют коммиты; удаление ветки их не стирает |
| Регрессии | Переносить вместе с тестами (фича 11), держать analyze 0/0 |

## 6. Резюме стратегии (одним абзацем)

`main` — чистая канва и будущая единственная релизная ветка. Расходящиеся ветки
несовместимы по истории и схеме БД, поэтому их **не сливаем**, а используем как
**каталог фич**: переносим нужные V2.5–V2.11 поверх свежего `main` отдельными
выверенными инкрементами (с продолжением нашей линии миграций и тестами), после
чего удаляем все ветки `claude/*`. На выходе — одна ветка `main` со всеми
фичами и зелёным CI.
