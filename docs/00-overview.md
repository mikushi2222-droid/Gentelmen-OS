# 00. Обзор проекта

## 1. Что мы строим

**Gentleman OS** — личное офлайн-приложение для одного пользователя. Это не
«цитатник из книг про стиль», а **личная система принятия решений** для мужчины:
что надеть, что купить, что улучшить, как выглядеть собраннее, как не ошибаться
с посадкой, цветами и дресс-кодом.

Главная ценность — **скорость ежедневного применения**. Для личного инструмента
выигрывает не ширина функционала, а быстрый ответ на бытовой вопрос «что делать
сегодня».

## 2. Почему именно так

Приложение строится как **offline-first, local-only, feature-first** продукт.
Для одного пользователя это даёт три преимущества:

1. **Приватность** — данные не покидают устройство.
2. **Быстродействие** — UI не ждёт сеть.
3. **Отсутствие лишней сложности** — нет сервера, аутентификации, синхронизации.

Flutter официально рекомендует offline-first подход для приложений, которым
важно работать без сети, и описывает разбиение на слои UI / logic / data с
чёткими правилами коммуникации между ними. Это хорошо ложится на личный инструмент.

## 3. Целевой пользователь

Один человек — владелец приложения. Конкретные параметры (рост, вес, талия,
объёмы) задаются в профиле и влияют на эвристики рекомендаций. Приложение
оптимизировано под крупную фигуру по умолчанию, но правила параметризованы и
подстраиваются под данные профиля.

## 4. Ключевое решение проекта

> Самое сильное решение здесь — **не ИИ, а локальное ядро с хорошими правилами.**
> Если оно качественное, приложение полезно даже без сети и без интеграций.

ИИ-слой — это будущее (v2+), а не фундамент. Фундамент — это цикл:

```
ввод данных → рекомендация → действие → сохранение результата
```

Пока этот цикл не работает идеально, всё остальное (синхронизация, чат-бот,
аналитика) преждевременно.

## 5. Что НЕ делаем (явные анти-цели)

- ❌ Сервер / backend
- ❌ Аутентификация и аккаунты
- ❌ Мультипользователь
- ❌ Социальная лента, чат для всех
- ❌ Маркетплейс
- ❌ Облачная синхронизация (в v1)
- ❌ Сложный чат-бот на первом этапе

Эти вещи только раздувают проект и мешают ядру.

## 6. Нефункциональные требования

### Производительность
- Быстрый холодный старт (цель < 2 с до интерактивного Dashboard).
- Основной UI никогда не блокируется сетью.
- Изображения гардероба кешируются и масштабируются (thumbnails).
- Длинные списки рендерятся лениво (`ListView.builder` / `SliverList`).

### Надёжность
- Приложение не теряет данные при перезапуске.
- Локальная БД имеет версионирование и миграции.
- Crash-safe записи (транзакции Drift).
- Экспорт/импорт всех данных обязателен.

### Приватность
- Все данные — локально.
- Внешние ключи — только если пользователь сам их добавит.
- Ничего не отправляется наружу без явного действия.

### Доступность
- Крупные заголовки.
- Контрастные цвета.
- Удобные touch targets (≥ 48dp).
- Тёмная тема по умолчанию.

### Тестируемость
- Unit-тесты для логики (use cases, scoring, репозитории).
- Widget-тесты для ключевых экранов.
- Golden-тесты для Dashboard / Wardrobe / Outfit Builder (по возможности).
- Тесты экспорта/импорта и миграций БД.

## 7. Метрики успеха (для себя)

- Время от запуска до «получить образ на сегодня» < 15 секунд.
- 100% данных переживают переустановку через экспорт/импорт.
- Любая рекомендация показывает «почему» (объяснимость).
- Ежедневное использование без раздражения от сложности.

## 8. Платформа

- **Android** (основная цель v1).
- Flutter позволяет позже расширить на iOS/desktop без переписывания ядра, но в
  v1 это не приоритет.
